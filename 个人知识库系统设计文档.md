## 1. 系统概述

个人知识库系统用于管理和复习计算机技术知识。系统分为两部分：
- **AI端**：我（Claude）在编写项目时读取知识库，结合你掌握的知识设计项目
- **人类端**：你通过网页浏览和复习知识库

---

## 2. 系统架构

```
┌─────────────────────────────────────────────────────┐
│                  个人知识库系统                      │
├─────────────────────┬───────────────────────────────┤
│    AI端（我）        │      人类端（你）              │
│                     │                               │
│  Markdown文件       │      网页（Flask + SQLite）    │
│  (knowledge/)       │                               │
│                     │                               │
│  - 项目设计前读取    │      - 浏览模式（主要）         │
│  - 代码中检测缺口    │      - 闪卡记忆（辅助）        │
│  - 更新知识点        │      - 测试模式（辅助）        │
│  - 更新后自动触发同步 │      - 只读访问               │
└─────────────────────┴───────────────────────────────┘
```

### 同步机制

```
AI端更新 Markdown
        │
        ▼
    AI端自动触发同步脚本
        │
        ▼
    解析 Markdown → 提取知识点
        │
        ▼
    增量更新 SQLite 数据库
    （内容字段以Markdown为准，复习字段保留）
        │
        ▼
    网页端显示最新数据
```

## 4. 数据模型

### 知识点表（knowledge_points）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER | 主键自增 |
| title | TEXT | 标题 |
| category | TEXT | 一级分类 |
| sub_category | TEXT | 二级分类 |
| content | TEXT | Markdown正文 |
| code_example | TEXT | 代码片段 |
| tags | TEXT | JSON数组 ["tag1","tag2"] |
| status | TEXT | 已学/待学/废弃 |
| mastery | INTEGER | 掌握程度1-5 |
| difficulty | INTEGER | 难度1-3 |
| level | TEXT | 入门/进阶/高级 |
| source | TEXT | 来源（书籍/视频/文档/项目/踩坑） |
| priority | TEXT | 高/中/低 |
| related_ids | TEXT | JSON数组 [1,3,5] |
| personal_notes | TEXT | 个人备注 |
| review_count | INTEGER | 复习次数 |
| last_reviewed_at | DATETIME | 上次复习时间 |
| next_review_date | DATETIME | 下次复习日期 |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 更新时间 |

### 字段分类（用于同步策略）

| 字段类型 | 包含字段 | 同步行为 |
|----------|----------|----------|
| 内容字段 | title、category、sub_category、content、code_example、tags、status、difficulty、level、source、priority、related_ids、personal_notes | 以Markdown为准 |
| 复习字段 | mastery、review_count、last_reviewed_at、next_review_date | 保留SQLite原值 |

### 状态流转规则

```
新建 → status=待学, mastery=0
首次学习 → status=已学, mastery=1
复习后：记住了 → mastery+1（最高5），没记住 → mastery-1（最低1）
内容过时 → 更新 updated_at + 修正 mastery
发现错误 → status=废弃（保留记录但不参与复习）
```

### status字段说明

| 值 | 含义 |
|-----|------|
| 已学 | 已掌握的内容，定期复习 |
| 待学 | 还没学到或需要补充，优先安排学习 |
| 废弃 | 内容过时或错误，保留记录但不参与复习 |

## 5. 建表DDL

### 主表

```sql
CREATE TABLE knowledge_points (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    title TEXT NOT NULL,
    category TEXT NOT NULL,
    sub_category TEXT NOT NULL,
    content TEXT NOT NULL DEFAULT '',
    code_example TEXT DEFAULT '',
    tags TEXT DEFAULT '[]',
    status TEXT NOT NULL DEFAULT '待学',
    mastery INTEGER NOT NULL DEFAULT 0,
    difficulty INTEGER DEFAULT 1,
    level TEXT DEFAULT '入门',
    source TEXT DEFAULT '',
    priority TEXT DEFAULT '中',
    related_ids TEXT DEFAULT '[]',
    personal_notes TEXT DEFAULT '',
    review_count INTEGER DEFAULT 0,
    last_reviewed_at DATETIME,
    next_review_date DATETIME,
    created_at DATETIME NOT NULL DEFAULT (datetime('now', 'localtime')),
    updated_at DATETIME NOT NULL DEFAULT (datetime('now', 'localtime'))
);
```

### 索引

```sql
-- 按分类快速查询
CREATE INDEX idx_category ON knowledge_points(category);
CREATE INDEX idx_sub_category ON knowledge_points(sub_category);

-- 按状态和mastery筛选（闪卡抽样用）
CREATE INDEX idx_status_mastery ON knowledge_points(status, mastery);

-- 按标签搜索
CREATE INDEX idx_tags ON knowledge_points(tags);
```

### 设计决策说明

| 决策 | 原因 |
|------|------|
| `tags` 和 `related_ids` 存 JSON 字符串 | SQLite 没有数组类型，存字符串用代码解析 |
| `status` 默认 `待学` | 新知识点默认待学状态 |
| `mastery` 默认 0 | 0 表示完全没学，权重最高 |
| `created_at` 和 `updated_at` 用 SQLite 函数 | 自动填充，不用应用层操心 |
| 索引覆盖抽样查询 | 闪卡按 status+mastery 抽样，索引加速 |

## 6. 闪卡/测试算法

### 核心原则

**不限制数量，不显示进度，随时可以打开随时退出。**

### 闪卡流程

```
打开闪卡
    │
    ▼
随机抽一个知识点
（权重：mastery低的优先，废弃知识点不出现）
    │
    ▼
展示：标题 + 正文（答案部分隐藏）
    │
    ▼
点"记住了" 或 "没记住"
    │
    ├─── 记住了 → mastery +1（最高5）
    └─── 没记住 → mastery -1（最低1）
    │
    ▼
自动加载下一个（无任何提示，直接跳到下一个）
```

**界面特点**：
- 只有"记住了"/"没记住"两个按钮
- 没有进度条，没有"已完成X个"的提示
- 关闭页面 = 结束本次复习，下次打开从头开始（随机）
- 不记录"本次复习了哪些"

### 测试流程（四选一）

```
打开测试
    │
    ▼
随机抽一个知识点作为题目
    │
    ▼
从同分类（sub_category）随机抽3个错误选项
    │
    ▼
展示：题目 + 4个选项（随机排列）
    │
    ▼
选择答案
    │
    ├─── 正确 → 显示绿色 + mastery +1
    └─── 错误 → 显示红色 + 正确答案 + mastery 不变
    │
    ▼
自动加载下一题（无任何提示）
```

**界面特点**：
- 同闪卡，答完直接跳下一题，无进度显示
- 关闭 = 退出，不记录"测了多少"

### 抽样算法

```python
# 闪卡抽样逻辑（伪代码）
所有知识点 = SELECT * FROM knowledge_points WHERE status != '废弃'
权重列表 = []

for 知识点 in 所有知识点:
    if mastery == 0: 权重 = 10   # 完全没学过，优先出现
    elif mastery <= 2: 权重 = 5  # 掌握度低，多复习
    elif mastery <= 4: 权重 = 2  # 掌握度中等
    else: 权重 = 1                # 掌握度高，少出现

    权重列表.append(权重)

# 按权重随机选一个
selected = random.choices(所有知识点, weights=权重列表, k=1)[0]
```

### 设计意图

- **不设每日目标**：有时间就打开，没时间就不打开，不要有心理负担
- **不跟踪本次复习**：每次打开都是新的随机开始，不记录"上次到哪了"
- **不显示进度**：没有"15/30"这种东西，看不到终点就不会焦虑

### 9.1 触发时机

- **开始新项目时**：检查技术栈，对照知识库设计方案

```

    │
    ▼

    │

    │
    ├─── 方案可行 → 直接用
    │
    └─── 有风险/不够用 → 给出多个方案和建议
                        │

    │

## 10. 同步机制

### 10.1 同步触发方式

**AI端自动触发**（主）：
- AI端更新 Markdown 文件后，自动运行同步脚本
- 减少手动操作的遗忘

**手动触发**（兜底）：
- 命令行运行 `python sync/sync_to_sqlite.py`
- 用于排查问题时

### 10.2 同步算法（增量同步）

```python
def sync():
    # 1. 遍历所有 Markdown 文件
    for md_file in all_markdown_files():
        point = parse_markdown(md_file)
        
        # 2. 查询 SQLite 中是否已存在
        existing = db.find_by_title_and_category(
            point.title, point.category, point.sub_category
        )
        
        if existing:
            # 3a. 已存在：更新内容字段，保留复习字段
            db.update_content_fields(existing.id, point)
            # mastery、review_count 等不更新
        else:
            # 3b. 新知识点：全量插入
            db.insert(point)
    
    # 4. 删除 orphan（SQLite 中存在但 Markdown 已删除的）
    for orphaned in db.find_not_in_markdown():
        db.delete(orphaned.id)
```

### 10.3 同步策略核心

| 原则 | 含义 |
|------|------|
| Markdown 是唯一数据源 | 所有内容字段以 Markdown 为准 |
| 复习数据保留 | mastery、review_count 等不在同步中覆盖 |
| 删除 orphan | Markdown 删除的内容，SQLite 也删除 |

### 10.4 Markdown 一般只添加不删除

实际使用中，Markdown 文件一般只添加新知识点，不删除旧内容。所以 orphan 删除情况很少发生，但保留这个机制以防万一。

## 12. 部署文档

### 12.1 部署方案

**方案A：本地运行 + Cloudflare Tunnel**

- 阶段一：本地运行（先跑通）
- 阶段二：加上 Cloudflare Tunnel（按需开启公网访问）

### 12.2 环境要求

- Python 3.9+
- Git

### 12.3 安装步骤

```bash
# 1. 克隆仓库
git clone https://github.com/<username>/knowledge-base.git
cd knowledge-base

# 2. 安装依赖
pip install -r web/requirements.txt

# 3. 初始化数据库（首次运行）
python sync/sync_to_sqlite.py

# 4. 启动服务
python web/app.py
# 浏览器打开 http://localhost:5000
```

### 12.4 公网访问（Cloudflare Tunnel）

```bash
# 安装 cloudflared（一次性）
# Windows: 下载 https://github.com/cloudflare/cloudflared/releases

# 启动隧道（每次访问时运行）
cloudflared tunnel --url http://localhost:5000

# 复制输出的公网URL（如 https://xxx.trycloudflare.com）
# 分享给同学即可访问你的知识库
```

## 14. 多用户架构

### 14.1 架构选择

**单人模式 + 公网只读访问**

### 14.2 行为说明

| 角色 | 行为 | 技术需求 |
|------|------|----------|
| 知识库所有者 | 本地更新 Markdown → 同步 → 查看/复习 | 完整功能 |
| 同一知识库的访客 | 通过公网URL查看知识库 | 只读功能 |
| 其他人 | 自己部署一套，用自己的知识库 | 完整功能 |

### 14.3 核心特点

- **不需要账号系统**：每个用户的知识库是独立的
- **不真正共享知识库**：A 的知识库和 B 的知识库互不影响
- **公网URL分享**：通过 Cloudflare Tunnel 暴露给指定的人
- **只读访问**：别人能看到你的内容，但不能修改
- **数据隔离**：每个人各自的 SQLite 数据库，互不干扰

## 16. 分享与扩展计划

### 16.1 分享对象

### 16.2 分享方式

通过 GitHub 仓库分享，同学克隆代码后本地运行：

```bash
git clone https://github.com/<username>/knowledge-base.git
cd knowledge-base
pip install -r web/requirements.txt
python web/app.py
```

### 16.3 分享时的初始内容

提供知识库模板，新用户获得：
- 完整的目录结构（分类文件夹）

- 每个分类 1-2 个示例知识点
- 完整的部署文档
